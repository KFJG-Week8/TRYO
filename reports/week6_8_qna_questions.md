# 6주차~8주차 QnA 질문 리스트

## 문서 목적
이 문서는 학생들과 QnA를 진행할 때, 단순히 "돌아간다"가 아니라 "핵심 로직을 이해하고 설명할 수 있는가"를 확인하기 위해 던질 수 있는 질문을 정리한 것이다.

질문은 정답 암기보다 다음을 확인하는 방향으로 구성했다.

- 왜 그렇게 설계했는가
- 현재 구조의 장단점을 알고 있는가
- 요구사항과 구현 사이의 차이를 인지하는가
- 다음 단계 개선 방향을 말할 수 있는가

## 1. 전체 구조 질문
1. 이 프로젝트를 한 문장으로 설명하면 무엇인가요?
2. `main()`에서 시작해서 요청 한 건이 응답으로 끝날 때까지 호출 흐름을 설명해보세요.
3. `server.c`, `sql.c`, `db.c`, `bptree.c`, `thread_pool.c`로 나눈 이유는 무엇인가요?
4. API 서버와 DB 엔진의 경계는 어디라고 생각하나요?
5. 이 프로젝트에서 가장 핵심이 되는 파일 하나만 고른다면 무엇이고, 왜 그런가요?

7. 지금 프로젝트에서 SQL 처리기 역할을 하는 함수는 무엇인가요?
8. 입력 SQL은 어디서 받아서 어디로 전달되나요?
9. 왜 `INSERT`와 `SELECT`만 지원하나요?
10. `UPDATE`, `DELETE`, `JOIN`을 넣지 않은 이유는 무엇인가요?
11. SQL parser와 DB 실행 로직을 분리한 이유는 무엇인가요?
12. `INSERT INTO users VALUES (1, 'bumsang', 25);`와 `INSERT INTO users name age VALUES 'kim' 20;`를 둘 다 지원하는 이유는 무엇인가요?
13. `SELECT id, name FROM users;` projection 기능은 왜 넣었나요?
14. `WHERE age = 20`을 왜 막았나요?
15. SQL parser가 가장 취약한 입력은 무엇이라고 보나요?
16. 지금 구조에서 SQL 텍스트 파일을 CLI로 받아 실행하려면 무엇을 추가해야 하나요?
17. 6주차 과제 산출물로 제출한다면 어떤 점이 부족하다고 인정해야 하나요?

## 마무리 코멘트
좋은 QnA는 학생을 몰아붙이는 시간이 아니라, 코드의 주도권이 학생에게 있는지 확인하는 시간이다. 그래서 질문은 "왜 그렇게 만들었는지", "무엇이 부족한지", "다음에 무엇을 바꿀지"를 말하게 만드는 것이 좋다.

이 프로젝트에서는 특히 다음 네 가지를 꼭 확인하면 된다.

1. 6주차 CLI 요구사항과 현재 API 서버 결과물의 차이를 인지하고 있는가
2. B+ 트리가 왜 `WHERE id = ?`에서만 쓰이는지 설명할 수 있는가
3. thread pool과 rwlock의 역할 차이를 설명할 수 있는가
4. 지금 구조의 한계와 개선 방향을 스스로 말할 수 있는가
